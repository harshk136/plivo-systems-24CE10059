The Flaky Network

System Architecture & Design Decisions

1. The Problem Space

Our goal was to transmit 160-byte real-time audio frames across a hostile UDP network capable of severe drops, delays, duplicates, and reordering. We had strict operational bounds: a 2.0x maximum bandwidth overhead budget, and a strict 1.0% maximum deadline-miss cap. This necessitated a careful balance of redundancy and latency.

2. Forward Error Correction (FEC) Protocol

We designed a custom wire protocol utilizing K=2 bitwise XOR Forward Error Correction. Appending the previous payload to every packet would have resulted in 324-byte packets (a 2.025x overhead), failing the grader checks.

  • The Math: The sender emits a standard 164-byte packet for every frame.
  • The Parity: For every second frame, it instantly generates and transmits a third packet containing the bitwise XOR of the current and preceding payloads.
  • The Result: Bandwidth overhead is mathematically locked at a highly efficient 1.54x (492 bytes every 2 frames), cleanly satisfying the 2.0x limit while mitigating single-packet drops.
  • Network Formatting: Sequences are processed using ntohl and htonl. Parity packets are flagged via the highest sequence bit (0x80000000), avoiding extra byte overhead.

3. Thread-Safe Jitter Buffer

To orchestrate reordered and duplicated packets, the receiver was completely decoupled into two asynchronous threads leveraging the C++ Standard Template Library.

  • The Receive Thread: A fast loop blocked on recvfrom. Packets are parsed, and missing data is instantaneously reconstructed in memory using the XOR parity algorithm the moment the adjacent packets arrive.
  • Memory Management: Sequence payloads are stored in a std::unordered_map, protected by a std::mutex, enabling O(1) lookups. Stale frames are actively aggressively pruned to prevent memory leaks.
  • The Playout Thread: A dedicated timing thread calculates the exact delivery deadline. It utilizes std::this_thread::sleep_until() to dispatch the reconstructed packets back to the harness player.

4. IPC Latency & Tuning

During initial testing, the jitter buffer encountered a 100% deadline-miss rate. The playout thread slept until the exact deadline, but inter-process communication latency across localhost UDP meant the Python grader recorded arrival slightly late.

The Fix: We shifted the playout schedule forward by 5ms. We subsequently stress-tested the system on unknown network topologies, successfully locking in a robust optimal delay.
